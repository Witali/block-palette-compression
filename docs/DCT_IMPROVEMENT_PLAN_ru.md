# План улучшения DCTBS2

Источник: [общий ChatGPT-чат](https://chatgpt.com/share/6a576088-76fc-83ed-9cf7-2d59e1693c69), последнее сообщение ассистента (`3245e917-aa94-4f56-b467-de671b971da7`).

Да. На таких низких битрейтах больший выигрыш даст уже не точная настройка одного DCT, а **адаптивный выбор способа кодирования каждого MCU**. Причём размер MCU можно оставить фиксированным, поэтому адресация не усложнится.

## 1. Перераспределять биты между яркостью и цветностью

Сейчас, например, режим 1 bpp использует:

```text
Y  = 16 байт
Cb =  8 байт
Cr =  8 байт
Итого 32 байта
```

Но для большинства фотографий яркость важнее цветности. Можно иметь несколько внутренних профилей:

```text
Профиль 0: Y16 + Cb8 + Cr8
Профиль 1: Y20 + Cb6 + Cr6
Профиль 2: Y22 + Cb5 + Cr5
Профиль 3: Y24 + Cb4 + Cr4
```

Общий размер остаётся 32 байта. В начале MCU хранится двухбитный номер профиля.

Адрес MCU остаётся простым:

```javascript
offset = headerSize + mcuIndex * 32;
```

Меняется только внутреннее смещение компонентов. На фотографиях режимы с большей долей яркости, вероятно, будут давать лучший RGB PSNR.

Для почти серых MCU можно использовать:

```text
Y28 + средний Cb 2 байта + средний Cr 2 байта
```

Это особенно эффективно для теней, облаков, бетона, металла и чёрно-белых участков.

## 2. Адаптивно выбирать размер преобразования

Один DCT 16×16 удобен для плавных областей, но плохо кодирует блок, внутри которого находятся несколько независимых объектов или резкая граница.

При том же размере MCU можно выбирать:

```text
режим 0: один DCT 16×16
режим 1: два DCT 16×8
режим 2: два DCT 8×16
режим 3: четыре DCT 8×8
```

Это один из наиболее перспективных вариантов.

- `16×16` — небо, стены, плавные градиенты;
- `16×8` — горизонтальные границы;
- `8×16` — вертикальные;
- `4 × 8×8` — текст, углы, мелкие объекты.

Кодер пробует все четыре варианта и выбирает минимальную ошибку. Общий бюджет, например 32 или 24 байта, остаётся тем же.

Это также сократит ringing: резкая граница будет проходить по границе преобразования или затронет меньший блок.

## 3. Разные преобразования для разных границ

DCT предполагает определённое симметричное продолжение блока. Для границ возле края блока иногда лучше подходят другие преобразования:

- DCT-II;
- DST;
- asymmetric DST, или ADST;
- горизонтально-направленное преобразование;
- вертикально-направленное;
- диагональное.

Можно хранить 2–3 бита номера преобразования. Например:

```text
0 — обычный DCT
1 — горизонтальный ADST
2 — вертикальный ADST
3 — диагональный профиль
```

ADST часто лучше представляет границу, расположенную возле одного края блока, и уменьшает волны без необходимости хранить много высоких частот.

## 4. Два масштаба коэффициентов вместо одного

Сейчас один масштаб применяется ко всем значениям блока:

```text
×1, ×2, ×3 или ×6
```

Но низкие и высокие частоты имеют разный диапазон. Лучше разделить их:

```text
scaleLow  — DC и первые низкочастотные AC
scaleHigh — остальные коэффициенты
```

Например:

```text
2 бита — scaleLow
2 бита — scaleHigh
```

Это улучшает точность важных низких частот и одновременно позволяет вместить редкие крупные высокочастотные значения.

Такая схема обычно эффективнее FP8, поскольку не тратит отдельную экспоненту на каждый коэффициент.

## 5. Неравномерная разрядность коэффициентов

Вместо двадцати коэффициентов по 5 бит можно хранить:

```text
DC                 10 бит
первые 4 AC         7 бит
следующие 8 AC      5 бит
остальные AC        3–4 бита
```

Низкие частоты сильнее влияют на MSE, поэтому им выгодно отдать больше точности.

Можно создавать несколько фиксированных битовых профилей:

```text
профиль A — больше точности низких частот;
профиль B — больше компонентов;
профиль C — для резких границ;
профиль D — для гладких областей.
```

Кодер выбирает профиль с минимальной ошибкой.

## 6. Обученные скалярные квантователи

Обычный signed integer предполагает равномерные уровни:

```text
−3, −2, −1, 0, 1, 2, 3...
```

Но распределение DCT-коэффициентов сильно сосредоточено возле нуля. Поэтому при 4–6 битах выгоднее использовать неравномерные уровни, например:

```text
0, ±1, ±2, ±3, ±5, ±8, ±12, ±18...
```

Декодер хранит таблицу из 16, 32 или 64 восстановленных значений:

```glsl
coefficient = coefficientCodebook[code];
```

Такие уровни можно подобрать алгоритмом Ллойда—Макса на большом наборе изображений. Это особенно полезно для 4-, 5- и 6-битных коэффициентов.

## 7. Векторное квантование групп коэффициентов

Вместо независимого хранения четырёх коэффициентов:

```text
c1, c2, c3, c4
```

можно хранить один индекс:

```text
8 бит → один из 256 заранее обученных векторов
```

Библиотека содержит:

```text
dictionary[index] = {c1, c2, c3, c4}
```

Это называется векторным квантованием. Коэффициенты коррелированы, поэтому общий индекс часто представляет их точнее, чем четыре независимых двухбитных или трёхбитных значения.

Практичная схема:

```text
группа 1: четыре самых низких AC → индекс 8 бит
группа 2: следующие четыре       → индекс 7 бит
группа 3: направленные частоты   → индекс 6 бит
```

На GPU это просто таблица или текстура со словарём.

## 8. Улучшить остаточный словарь

Текущий словарь границ можно сделать значительно эффективнее.

Вместо вручную созданных полос следует обучить его на реальных остатках:

```text
residual = original − decoded
```

Затем:

1. собрать миллионы остаточных блоков;
2. нормализовать их амплитуду;
3. кластеризовать K-means;
4. сохранить 64–256 центров кластеров;
5. дообучить центры с учётом реальной MSE после квантования.

Полезно иметь несколько библиотек:

```text
словарь 0 — вертикальные и горизонтальные границы
словарь 1 — диагонали
словарь 2 — углы и тонкие линии
словарь 3 — фотографические текстуры
```

Ещё лучше — разрешить сумму двух небольших шаблонов:

```text
residual =
    amplitude1 × dictionary1[index1] +
    amplitude2 × dictionary2[index2]
```

Это residual vector quantization. Даже два индекса по 5–6 бит могут оказаться эффективнее нескольких обычных DCT-компонентов.

## 9. Предсказание блока до DCT

Вместо кодирования самого блока можно сначала предсказать его структуру, а DCT применить только к ошибке.

Возможные предсказатели:

```text
DC       — блок одного цвета;
vertical — продолжение верхней границы;
horizontal — продолжение левой границы;
planar   — плавный двумерный градиент;
diagonal — продолжение диагонали.
```

Затем:

```text
residual = original − prediction
```

и уже residual кодируется DCT.

Это один из самых сильных методов, но он требует пикселей соседних MCU. Для произвольного доступа к одной координате придётся дополнительно декодировать верхний и левый блоки либо периодически вставлять независимые опорные MCU.

Компромисс:

- каждая четвёртая строка и колонка MCU независима;
- между ними разрешено предсказание;
- random access требует декодировать не более нескольких соседних MCU.

## 10. Улучшить работу с цветностью

При низком битрейте часть ошибки может идти не от DCT, а от Cb/Cr.

Возможные улучшения без увеличения файла:

- более качественный фильтр уменьшения цветности;
- edge-directed upsampling;
- восстановление цветности с учётом границ яркости;
- выбор между 4:2:0, 4:2:2 и почти постоянной цветностью на MCU;
- переключение между YCbCr и YCoCg.

Например, при восстановлении Cb/Cr не интерполировать через резкую границу яркости:

```text
если Y слева сильно отличается от Y справа,
не смешивать их цветность
```

На цветных контурах это может заметно поднять RGB PSNR.

## 11. Оптимизировать по финальному RGB, а не по DCT-ошибке

Сейчас удобно оценивать ошибку в пространстве коэффициентов. Для ортонормированного DCT это точно соответствует ошибке одной компоненты, но не учитывает:

- преобразование YCbCr → RGB;
- интерполяцию цветности;
- clipping в диапазон 0…255;
- словарную коррекцию;
- соседние блоки.

Лучший кодер должен для нескольких финальных кандидатов:

1. распаковать MCU;
2. преобразовать его в RGB;
3. вычислить реальный RGB MSE;
4. выбрать вариант с максимальным PSNR.

Это медленнее только при кодировании. Декодер и формат не усложняются.

## 12. Компенсация систематической ошибки

После квантования восстановленные коэффициенты могут иметь небольшой систематический сдвиг. Вместо:

```text
restored = code × scale
```

можно использовать таблицы:

```text
restored = reconstructionTable[frequency][code]
```

Таблица обучается как среднее исходное значение для каждого кода. Это даёт оптимальный по MSE уровень восстановления и ничего не добавляет в файл.

## Что я бы реализовал следующим

По ожидаемой эффективности:

1. **Адаптивное разбиение 16×16 / 16×8 / 8×16 / 8×8.**
2. **Несколько вариантов распределения байтов между Y, Cb и Cr.**
3. **Два масштаба: для низких и высоких частот.**
4. **Обученные неравномерные квантователи для 4–6-битных значений.**
5. **Обученный словарь реальных residual-блоков вместо ручных полос.**
6. **Финальный выбор режима по RGB PSNR.**

Наиболее разумный следующий эксперимент — совместить пункты 1 и 2. Они не требуют внешней большой библиотеки и могут дать более заметный выигрыш, чем текущий словарь границ, особенно на режимах 1 и 0,75 бит/пиксель.
